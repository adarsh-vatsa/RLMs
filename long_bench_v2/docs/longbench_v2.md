# LongBench-v2 Cache Dataset Preparation

This folder contains utilities for preparing a modified LongBench-v2 dataset for cache-route evaluation. The workflow exports the original JSON dataset to an observable CSV, generates semantic-question rewrites, generates setup/knowledge workload rows, and combines the reviewed outputs into a final suite.

The original dataset remains the source of truth for full context text. The CSV files intentionally do not copy the `context` field because many LongBench-v2 contexts are too large for comfortable spreadsheet review.

## Summary of Steps

- Step 1: Extract rows from `benchmark_data/long_bench_v2/data.json` and write to `benchmark_data/long_bench_v2/data.csv`
- Step 2: `row_type=original` samples in `benchmark_data/long_bench_v2/data.csv` are sent to the Codex SDK to generate rewritten questions for semantic hit cases. Output is saved to `benchmark_data/long_bench_v2/data_semantic_codex.csv`.
- Step 3: Unique original contexts in `benchmark_data/long_bench_v2/data.csv` are sent to the Codex SDK to generate setup questions for knowledge workload rows. Output is saved to `benchmark_data/long_bench_v2/data_knowledge_codex.csv`.
- Step 4: Combine `data.csv`, `data_semantic_codex.csv`, and `data_knowledge_codex.csv` into `benchmark_data/long_bench_v2/data_cache_suite.csv`.

## STEP 1 - Export Original And Exact Rows

Use `long_bench_v2/export_csv.py` to create a metadata CSV for inspecting the original LongBench-v2 JSON dataset and preparing exact-match cache rows.

The exporter does not modify `benchmark_data/long_bench_v2/data.json`. It reads the JSON list and writes a flat CSV with `_id` renamed to `source_id`. It does not copy the full `context` field into the CSV; rows reference the original JSON through `source_id` and `context_id`.

By default, each original LongBench-v2 row produces two CSV rows:

- `row_type=original`: first-use row, expected cache miss.
- `row_type=exact`: duplicate row, expected exact cache hit after its original row has run.

Semantic and knowledge rows are not generated by this exporter. Use the Codex SDK workflows below after creating `data.csv`.

### Command (STEP 1 - RUN THIS)

```bash
python long_bench_v2/export_csv.py \
  --input-path benchmark_data/long_bench_v2/data.json \
  --output-path benchmark_data/long_bench_v2/data.csv
```

To export only original rows:

```bash
python long_bench_v2/export_csv.py \
  --input-path benchmark_data/long_bench_v2/data.json \
  --output-path benchmark_data/long_bench_v2/data_original_only.csv \
  --no-include-exact
```

### Columns

The CSV columns are `case_id`, `source_id`, `row_type`, `is_scored`, `setup_case_id`, `context_id`, `expected_cache_type`, `expected_from_cache`, `depends_on_case_id`, `domain`, `sub_domain`, `difficulty`, `length`, `question`, `choice_A`, `choice_B`, `choice_C`, `choice_D`, and `answer`. The `context_id` is a short SHA-256 prefix of the original context text, which helps identify repeated contexts without copying the large context into spreadsheet cells.

Use the CSV for observability and filtering. The original JSON remains the canonical source for full context text.

Route fields are metadata for post-run analysis. Benchmark runners should not use `row_type`, `expected_cache_type`, or `expected_from_cache` to choose a route; execution should receive only the context, question, and choices.

## STEP 2 - Generate Semantic Question Rewrites

Use `long_bench_v2/generate_semantic_questions.ts` to create semantic rows from the `row_type=original` rows in `benchmark_data/long_bench_v2/data.csv` and rewrite their `question` values with the official Codex TypeScript SDK. The script does not use OpenRouter and does not load the full LongBench-v2 context JSON.

For each original row, the generator creates a semantic row with:

- `row_type=semantic`
- `expected_cache_type=semantic`
- `expected_from_cache=true`
- `depends_on_case_id=<source_id>__original`

The generator preserves the original metadata and choices, changes only the route metadata listed above, and writes a separate semantic CSV by default so the generated questions can be reviewed before merging them back into the main benchmark CSV.

### Setup

Install the SDK and TypeScript runner from the benchmark tool folder:

```bash
cd long_bench_v2
npm install
```

The script uses the official Codex TypeScript SDK package, `@openai/codex-sdk`. Make sure your OpenAI/Codex authentication is configured in the environment before running generation.

### Pilot Run

Run a small pilot first:

```bash
npm run generate-semantic-questions -- \
  --limit 10
```

When `--limit 10` is set, the output CSV contains only those 10 generated semantic rows. Full runs without `--limit` write the complete generated semantic set.

Default pilot outputs:

- `benchmark_data/long_bench_v2/data_semantic_codex.csv`
- `benchmark_data/long_bench_v2/data_semantic_codex_audit.jsonl`

Review both files before running all rows. The CSV shows the generated questions in spreadsheet form. The audit log records source ids, original questions, generated questions, raw Codex responses, and validation warnings.

### Full Run (STEP 2 - DIRECTLY RUN THIS IF YOU ALREADY INSTALLED REQUIRED PACKAGES)

After reviewing the pilot output and audit log, run the full file:

```bash
npm run generate-semantic-questions -- \
  --model gpt-5.5 \
  --reasoning-effort high \
  --delay-ms 100 \
  --no-keep-existing-on-error \
  --force
```

IMPORTANT: Use `--force` only when you intentionally want to overwrite the previous generated CSV and audit log.

### Useful Options

- `--source-csv PATH`: original/exact CSV used to recover original questions by `source_id`. Default: `benchmark_data/long_bench_v2/data.csv`.
- `--output-csv PATH`: generated semantic CSV. Default: `benchmark_data/long_bench_v2/data_semantic_codex.csv`.
- `--audit-path PATH`: JSONL audit log path. Default: `benchmark_data/long_bench_v2/data_semantic_codex_audit.jsonl`.
- `--model MODEL`: Codex model to use. If omitted, the Codex SDK/CLI default is used.
- `--reasoning-effort EFFORT`: reasoning effort for the selected model. Valid values are `minimal`, `low`, `medium`, `high`, and `xhigh`. If omitted, the Codex SDK/CLI default is used.
- `--start-index N`: resume from a zero-based input row index.
- `--limit N`: generate only N semantic rows.
- `--delay-ms N`: wait between Codex calls.
- `--max-retries N`: retry failed rows.
- `--no-keep-existing-on-error`: fail the run instead of keeping the original question when generation fails.

### Validation Behavior

The generator asks Codex to return strict JSON with a single `rewritten_question`. It rejects empty rewrites, retries rows where Codex includes answer choices inside `rewritten_question`, and logs warnings when important quoted strings or numeric tokens from the original question are missing. If Codex returns the original question unchanged, the script logs `status=codex_unchanged` so the row can be reviewed before it is used as a semantic cache case.

This validation is intentionally conservative. The output should still be manually reviewed before it becomes part of the benchmark suite.

## STEP 3 - Generate Knowledge Questions

Use `long_bench_v2/generate_knowledge_rows.ts` to create setup and knowledge rows from only the `row_type=original` rows in `benchmark_data/long_bench_v2/data.csv`. The script ignores exact, semantic, setup, and knowledge rows.

For each unique original `context_id`, the generator asks Codex for one open-ended setup question. It sends only compact metadata and representative original questions to Codex, not the full long context. For each original row, it also creates one scored `row_type=knowledge` row that points to that context's setup row.

Setup rows use:

- `row_type=setup`
- `is_scored=false`
- `expected_cache_type=miss`
- `expected_from_cache=false`
- blank choices and answer

Knowledge rows use:

- `row_type=knowledge`
- `is_scored=true`
- `setup_case_id=<context_id>__setup`
- `expected_cache_type=knowledge`
- `expected_from_cache=true`
- `depends_on_case_id=<context_id>__setup`

### Full Run (STEP 3 - YOU CAN LIMIT CONTEXTS FOR PILOT RUNS IF NEEDED)

```bash
cd long_bench_v2
npm run generate-knowledge-rows -- \
  --model gpt-5.5 \
  --reasoning-effort high \
  --delay-ms 100 \
  --force \
  --limit-contexts 5
```

Default outputs:

- `benchmark_data/long_bench_v2/data_knowledge_codex.csv`
- `benchmark_data/long_bench_v2/data_knowledge_codex_audit.jsonl`

Useful options:

- `--source-csv PATH`: original/exact CSV used to recover original rows by `context_id`. Default: `benchmark_data/long_bench_v2/data.csv`.
- `--output-csv PATH`: generated setup/knowledge CSV. Default: `benchmark_data/long_bench_v2/data_knowledge_codex.csv`.
- `--audit-path PATH`: JSONL audit log path. Default: `benchmark_data/long_bench_v2/data_knowledge_codex_audit.jsonl`.
- `--model MODEL`: Codex model to use. If omitted, the Codex SDK/CLI default is used.
- `--reasoning-effort EFFORT`: reasoning effort for the selected model. Valid values are `minimal`, `low`, `medium`, `high`, and `xhigh`.
- `--limit-contexts N`: generate setup rows for only the first N unique original contexts and knowledge rows only for those contexts.
- `--delay-ms N`: wait between Codex calls.
- `--max-retries N`: retry failed context setup generation.

## STEP 4 - Combine Rows

Use `long_bench_v2/combine_csv.py` after reviewing generated CSV files. This writes a separate combined file and does not edit `benchmark_data/long_bench_v2/data.csv`. `--semantic-csv-path` and `--knowledge-csv-path` are optional; only pass the files you want included.

For the main cold-start suite, include original/exact rows from `data.csv` and semantic rows only:

```bash
python long_bench_v2/combine_csv.py \
  --base-csv-path benchmark_data/long_bench_v2/data.csv \
  --semantic-csv-path benchmark_data/long_bench_v2/data_semantic_codex.csv \
  --output-path benchmark_data/long_bench_v2/data_cache_suite.csv
```

The combined file contains the original and exact rows from `data.csv`, followed by semantic rows from `data_semantic_codex.csv`. Do not pass `--knowledge-csv-path` for the main cold-start comparison.

For a later prewarm/knowledge experiment, pass the knowledge file explicitly:

```bash
python long_bench_v2/combine_csv.py \
  --base-csv-path benchmark_data/long_bench_v2/data.csv \
  --semantic-csv-path benchmark_data/long_bench_v2/data_semantic_codex.csv \
  --knowledge-csv-path benchmark_data/long_bench_v2/data_knowledge_codex.csv \
  --output-path benchmark_data/long_bench_v2/data_cache_suite_with_knowledge.csv
```

## STEP 5 - Sample A Smaller Test Suite

Use `long_bench_v2/sample_csv.py` to create a smaller balanced CSV for cheap local testing. The sampler chooses shared `source_id` values, so a sample size of 10 keeps the matching 10 `original`, 10 `exact`, and 10 `semantic` rows instead of unrelated rows from each group.

```bash
python long_bench_v2/sample_csv.py \
  --input-path benchmark_data/long_bench_v2/data_cache_suite.csv \
  --output-path benchmark_data/long_bench_v2/data_cache_suite_sample.csv \
  --sample-size 2 \
  --seed 0
```

Useful options:

- `--sample-size N`: number of rows to keep per row type. Default: `10`.
- `--row-types TYPES`: comma-separated row types to sample together by `source_id`. Default: `original,exact,semantic`.
- `--seed N`: random seed for reproducible samples. Default: `0`.
